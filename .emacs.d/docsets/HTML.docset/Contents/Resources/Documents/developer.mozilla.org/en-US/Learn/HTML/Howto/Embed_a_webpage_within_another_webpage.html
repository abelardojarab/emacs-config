    Embed a webpage within another webpage

    by 3 contributors:

    We're going over how to embed one webpage in another with as little risk as possible.

    Prerequisites: You need to be familiar with how to create a basic HTML document.
    Objective: Review the possible complications of nesting webpages, and learn how to nest webpages with less risk.

    Webpage nesting, then and now

    Nesting webpages sounds odd and unnatural, but we've been doing it since the very start of the Web. Back then, people were using painfully slow modem connections (56K or even less). To shorten the download time, we broke webpages up into small documents called frames, embedded in a master document called a frameset. Unfortunately, in practice HTML frames created more problems than they solved, and the whole "frames and frameset" idea has gone away now that AJAX has come on the scene.

    There are still valid reasons to nest webpages. When your webpage must include user-generated or third-party content (like ads), you can sandbox the content inside an HTML5 iframe for enhanced security. In advanced scenarios, nesting webpages can also improve performance by giving the browser smaller DOM trees to process.

    At the same time, nesting webpages still comes with significant security, performance, and accessibility consequences. Make sure you understand the risks and carefully weigh how you can best serve your customers.

    Basic nesting

    Most of the time, you'll want <iframe> when you're nesting webpages. Here's what you need to consider for getting started:

    • Where's the document I'm embedding? Put the URL into the src attribute.
    • How much space should the document take up in the parent webpage? Supply this information with width and height.
    • What happens if a user agent doesn't support iframes? Provide fallback HTML content (e.g., a link to the document you're embedding) between the opening and closing <iframe> tags.
    • Don't forget the highly advisable sandbox attribute, which requests heightened security.

    You'll end up with something like this:

    <iframe 
 src="https://developer.mozilla.org/en-US/docs/Glossary"
 width="100%" height="500"
 sandbox>
  <p>
    <a href="https://developer.mozilla.org/en-US/docs/Glossary">
      Fallback link for browsers that, unlikely, don't support frames
    </a>
  </p>
</iframe>

    In order to improve speed, It's a good idea to set the iframe's src attribute with JavaScript after the main content is done loading. This makes your page usable sooner and decreases your official page load time (an important SEO metric). Remember, you don't get extra browser-server connections just because there's several webpages' worth of stuff to load.

    If you don't like the default thick border around the <iframe>, you're welcome to specify border: none; in your CSS.

    Keep yourself safe

    Remember our ominous warnings about security issues? Browser makers and Web developers have learned the hard way that iframes are a common attack vector for hackers looking to maliciously modify your webpage, or trick people into doing something they shouldn't do.

    Clickjacking is one kind of iframe attack, where hackers put an invisible iframe into your document and capture users' interaction with your website. This is a common way to mislead users or steal steal sensitive data.

    From simplest to hardest, we'll cover some security measures you can take to protect yourself:

    Don't embed third-party content

    Sometimes you have to, but you can save yourself a lot of headaches if you don't. Here's the rule of thumb for Web security: "You can never be too cautious. If you made it, double-check it anyway. If someone else made it, it's dangerous."

    Even besides security, be aware of intellectual property issues. Most content is copyrighted, offline and online, even content you might not expect (for example, most images on Wikimedia Commons). Never display content on your webpage unless you own it or unless the owners have given you written, unequivocal permission. Penalties for copyright infringement are severe. Again, you can never be too cautious.

    If the content is licensed, you must obey the license terms. For example, our content at MDN is licensed under CC-BY-SA. That means, you must credit us properly when you quote our content, even if you make substantial changes.

    Use HTTPS

    HTTPS is the encrypted version of HTTP. You should consider using HTTPS whenever possible:

    1. HTTPS reduces the chance that remote content has been tampered with in transit,
    2. HTTPS prevents embedded content from accessing content in your parent document, and vice versa.

    Of course security certificates are not cheap, so if you can't get one, you may serve your parent document with HTTP. However, because of the second benefit of HTTPS above, no matter what the cost, you must never embed third-party content with HTTP. (Best case scenario, your user's Web browser will give them a scary warning.)

    Always use the sandbox attribute

    You want to give attackers as little power as you can. Therefore, give embedded content only the permissions indispensable for doing its job. Of course, this applies to your own content, too.

    Unsandboxed content can do way too much (scripting, forms, popup windows...). If at all possible, impose all available restrictions by using sandbox with no parameters.

    If you must, add permissions back one by one (inside the quote marks of sandbox=""). But never add both allow-scripts and allow-same-origin, since then the embedded content can turn off sandboxing altogether.

    Sandboxing provides no protection if attackers can fool people into visiting malicious content directly (outside an iframe). If there's any chance that certain content may be malicious (e.g., user-generated content), please serve it from a different domain than your main site.

    Line up communication between content

    In some cases it can be very convenient to allow some communication between an iframe content and the host document. It's really easy to handle with a bit of JavaScript. If you have properly sandboxed your iframe, neither the iframe nor the parent document can directly access the other content's DOM (without sandboxing, it is possible and amazingly insecure). The postMessage API is the only safe way for the two pieces of content to communicate.

    Configure CSP directives

    CSP provides a set of HTTP Headers designed to improve the security of your HTML document. When it comes to iframe, make sure you configure your server to send an appropriate X-Frame-Options  header. This can prevent other websites from embedding your content in their webpages (which would set the stage for clickjacking and a host of other attacks). Please read Frederik Braun's post for more background.

    Sandboxing HTML code

    We talked before about how you must sandbox all embedded content, even your own content. Often it's beneficial to go a step further and to break up your webpage into several sandboxed iframes, each with minimum privileges, all coordinated by the main document. (Mike West discusses at further length how privilege separation can provide significant security benefits. When used carefully with JavaScript, iframes can also improve performance by giving browsers smaller DOM trees to process at one time.)

    Armed with the two new iframe attributes seamless and srcdoc, you can embed an HTML code snippet just as readily as an HTML document.  You can support legacy browsers by fallback URL with src (possibly a DATA URI). Here's a basic example:

    <iframe
 sandbox
 seamless
 src="fallback.html"
 srcdoc="
   <p>
     I want this paragraph inside a sandbox.
   </p>
 ">
  Fallback for browsers that can't handle iframes
</iframe>

    Within the srcdoc attribute, you must escape all quotes (use &quot;) and double-escape all ampersands (use &amp;amp; for a single ampersand).
