    Add audio or video content to a webpage

    by 3 contributors:

    Here's how to provide a first-class audio and video experience to everyone, regardless of how they choose to consume media.

    Prerequisites: You need to be familiar with creating basic HTML documents.
    Objective: Learn how to embed audio and video in your page.

    Audio and video on the web

    Web developers have been trying to use video and audio on the Web since forever. Following much experimentation, HTML5 offers the best solution yet, with the help of some shiny new JavaScript APIs.

    In this article, we explain how to embed audio and video files in your HTML documents. We're assuming you already have audio and video files, because producing those files takes a completely different skillset.

    Before you begin here, you should also know that there are a quite a few OVPs (online video providers) like YouTube, Dailymotion, and Vimeo. OVPs offer a convenient way to host and consume videos, so you don't have to worry about the enormous bandwidth consumption. OVPs even usually offer user-friendly JavaScript scripts for embedding video in your own webpages. If you go that route, you can avoid some of the difficulties we discuss in this article.

    Quick start

    The easiest way to embed a media file in a webpage is to use the <audio> element for sound files and the <video> element for videos, like this:

    <audio src="example.ogg" controls></audio>
<video src="example.webm" width="375" height="280" controls></video>

    Those two HTML elements come with some convenient attributes to control their behaviors:

    width and height (video only)
    You can control the video size either with these attributes or with CSS. In all cases, videos retain their native width-height ratio (black strips take up the empty space in the video element).
    controls

    Users must be able to control video and audio playback (it's especially critical for victims of epilepsy). You must either use controls to request the native control interface, or build your own interface using the appropriate JavaScript API. At a minimum, the interface must include a way to start and stop the media, and to adjust the volume.

    autoplay
    This attribute makes the audio or video start playing right away, while the rest of the page is loading. Be careful, because users can find it quite annoying.
    loop
    This attribute makes the video and/or audio start playing again whenever it finishes. Use this also with care.
    muted
    This attribute plays the media with sound turned off by default.
    poster (video only)
    This attribute provides the URL of an image displayed before the video begins.

    Conquering codec obstacles

    The problem: browsers differ in codec support

    Codecs (like Vorbis or H.264) convert sound and video into binary digits and back. If you're missing the right codec, the binary data is just gibberish.

    Container format (like Ogg) refers to how sound data, metadata, and/or image data are packaged into a file. Container format support is its own issue, but systems usually support the container formats typical for the codecs they support. (Ogg tends to go with Vorbis audio and Theora video. WebM often packages Vorbis audio with VP8/VP9 video; MP4 packages AAC audio and H.264 video.)

    Unfortunately, browsers don't all support the same codecs, so you may have to provide several files for each media production.

    Which formats should I offer?

    MP3 (for audio) and MP4/H.264 (for video) are both widely supported. However, American patents cover MP3 until at least 2017, and H.264 until 2027 at the earliest. Many people avoid restricted software, and you can only retain those viewers if you also offer your media in free formats (like Ogg Vorbis for audio and WebM for video).

    Ideally, like Wikipedia, you'd provide files only in open formats, but Safari and Microsoft Edge (not to speak of Internet Explorer) don't support them. That leaves another large minority out in the cold.

    For more details, please see our audio codec compatibility table and our audio-video codec compatibility table.

    So how do I offer the same content in several formats?

    HTML provide de convinient <source> element you can use with an src attribute. Don't include src in the <video> or <audio> tag itself, because src overrides the <source> elements. Here's an example:

    <audio controls>
  <source src="example.mp3" type="audio/mpeg">
  <source src="example.ogg" type="audio/ogg">
</audio>

    Remind to include the type attribute. Otherwise, browsers will load and try each file, taking even more time and resources. type lets browsers skip immediately whichever files they can't play (if need be, all of them). Our article on supported media formats contains some common MIME types.

    Transcripts & subtitles

    "Transcribe" and "transcript" mean to write down spoken words as text.

    Many people cannot or don't want to consume audio and video content (for example, in noisy environments, or in the library). To serve these visitors, both as plain text and as a text track (i.e., as text that appears at various points in the video). It takes some time to produce, but it's worth the effort.

    Text tracks also help you with SEO, since search engines especially thrive on text. Text tracks allow search engines even to link directly to a spot partway through the video.

    Text tracks

    The text strings displayed during the video are called cues. Here are the most common kinds of cues:

    subtitles
    translations of foreign material
    captions
    synchronized transcriptions of dialog or descriptions of significant sounds (to let people watch the film on mute)
    timed descriptions
    text for conversion into audio (to serve the sight-challenged)

    HTML lets you include text tracks with audio and video content. Doing so is quite simple:

    1. Write one or more WebVTT text files. A WebVTT file contains the cues and specifies when they should appear and disappear (this is where having a good transcript helps).
    2. Link to the WebVTT text file(s) with the <track> element. <track> occurs within <audio> or <video>, but after all <source> elements. Use the kind attribute to specify whether the cues are subtitles, captions, or descriptions. Further, use srclang to tell the browser what language you wrote the subtitles in.

    Here's an example:

    <video controls>
    <source src="example.mp4" type="video/mp4">
    <source src="example.webm" type="video/webm">
    <track kind="subtitles" src="subtitles_en.vtt" srclang="en">
</video>

    For more details, please read Adding captions and subtitles to HTML5 video.

    Fallback for legacy browsers

    You can add fallback content for pre-HTML5 browsers immediately before the closing tag. For example, you can add a download link:

    <video src="example.webm" controls>
    <track kind="captions" src="captions.vtt" srclang="en">
    <a href="example.webm">Download the video</a>
</video>

    A download link is not user-friendly (especially on mobile), but maybe it's better than just telling people they have a dated browser. There are more advanced fallback techniques, but a dedicated third-party script might bring you the most satisfaction (like the scripts provided by OVPs).
