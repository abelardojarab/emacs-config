    Divide a webpage into logical sections

    by 4 contributors:

    We'll go over how to structure your document so it makes sense to you and your visitors, including people using assistive technologies.

    Prerequisites: You need to be familiar with how to create a basic HTML document.
    Objective: Learn how to structure your document using semantic tags.

    Basic sections

    Websites can look pretty different, but tend to share similar components:

    header
    Usually a big strip across the top with a big heading and/or logo. This is where the main common information about a website usually stays from one webpage to another.
    navigation bar
    links to the site's main sections; sometimes menus or tabs. Like the header it is often invariable content from one webpage to another. Many web designers consider the navigation bar to be part of the header rather than a individual component but that's not a requirement.
    main content
    A big area in the center with content unique to a given webpage.
    sidebar
    Some peripheral info, links, quotes, ads. Usually this is contextual to what is in the main content, but there are also cases where you'll find some recurring elements like a secondary navigation system.
    footer
    fine print, copyright, contact info… It's a place to put common information (like inside the header) but usually that information is not critical or secondary to the website itself. It is also sometimes used for SEO purposes by providing quick access to deeper parts of the website.

    HTML to the rescue

    The previous diagram uses visual cues, like green, to indicate that an area is acting as header, nav bar, content area, sidebar, or footer. Unfortunately, visuals aren't the whole story. A visually challenged person, like a colorblind person who represent 8% of the world population, can benefit from concepts like "nav bar" and "main content", where "Larger type" and "light green" aren't helpful.

    In your HTML code, you can mark up sections based on their functionality. That's what we're doing in this article. To learn how to add the visual cues, please take a look at our CSS tutorial.

    To perform such semantic mark up, HTML provides dedicated tags that you can use like this:

    • header: <header>
    • navigation bar: <nav>
    • main content: <main> plus a mix of <article>, <section>, and <div> to create sub-sections
    • sidebar: <aside>
    • footer: <footer>

    For the navigation bar, it's quite a common pattern to create an unordered list of links inside <nav> tags. Chris Coyier sums up the pros and cons of using lists vs. bare links.

    Which element for which purpose

    If you face a different use case, it's good to know the overall meaning of all the HTML sectioning elements:

    • <main> is for content unique to this page. Use <main> only once per page. <main> goes directly inside <body>, and, as much as possible, shouldn't be nested within other elements.
    • <article> encloses a block of related content that makes sense on its own without the rest of the page (e.g. a single blog post).
    • <section> is like <article>, but depends on the rest of the page as context. You can use sections for grouping articles or for breaking up an article. It's considered best practice to begin each section with a heading.
    • <aside> contains content that is not directly related to the main content but can provide aside information indirectly related to it (e.g. Glossary entries, Author biographiy, related links, etc.).
    • <header> represents a group of introductory content. If it is a child of <body> it defines the global header of a webpage, but if it's a child of an <article> or <section> it defines a specific header to that specific section (which should not be confused with sections' heading).
    • <footer> represents a group of ending content. Based on it's position it behaves like the <header> element.
    • <div> is an element you can use, preferably with a suitable class, when nothing more specific is available. DIVs are a non-semantic division of your document, which is convenient to manipulate specific parts of your document as one entity using CSS or JavaScript.

    Warning: DIVs are so very convenient to use that it's easy to use them too much. As they carry no semantic, they just clutter your HTML code. Take care to use them only when there is no better semantic solution and try to reduce their usage to the minimum otherwise you'll have a hard time updating and maintaining your documents.

    Example

    In terms of HTML code, the above diagram could looks like this

    <!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="utf-8">
  <title>HTML Document</title>
</head>
<body>

  <!--
  Here is our main header that is used accross
  all the pages of our website
  -->
  <header>
    header

    <!--
    Even is it's not mandatory, it's common
    practice to put the main navigation menu
    within the main header
    -->
    <nav>
      <ul>
        <li><a href="#">Navigation</a></li>
        <li><a href="#">Who we are</a></li>
        <li><a href="#">What we do</a></li>
        <li><a href="#">Contact</a></li>
      </ul>

      <!--
      Search form is another non-linear way
      to navigate through a website.
      -->
      <form>
        <input type="search" name="q" placeholder="Search query">
        <input type="submit" value="Go!">
      </form>
    </nav>
  </header>

  <!--
  Here is our page main content
  -->
  <main>

      <h2>subsection</h2>

    Introducing ARIA roles

    HTML is originally for specifying document structure. We've come to use HTML as a foundation for Web applications.

    ARIA is a well-supported way to tell assistive technology that an HTML element has Web applications functionality.

    To the extent some HTML5 elements like <nav> convey such information, which make ARIA role attributes redundant. However, if you're targeting technology that doesn't support HTML5 or want to use a non-semantic element (such as <div>) , consider adding ARIA roles to HTML5 tags, like this:

    • <main role="main">
    • <header role="banner">
    • <nav role="navigation">
    • <aside role="complementary">
    • <footer role="contentinfo">

    To learn more about ARIA, please refer to our documentation.
