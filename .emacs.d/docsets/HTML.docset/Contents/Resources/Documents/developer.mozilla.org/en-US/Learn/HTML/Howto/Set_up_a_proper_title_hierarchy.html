    Set up a proper title hierarchy

    by 4 contributors:
    Show Sidebar

    We'll go over how to put headings in your document so your readers can get the answers they're looking for faster.

    Prerequisites: You need to be familiar with how to create a basic HTML document.
    Objective: FInd out why and how to insert headings into your document.

    Why use headings?

    Headings are the backbone of any text content, rather than providing long explanations, let's take a simple example: Which of these two documents would you rather read?

    Quite enlightening, isn't it?

    So as you can see, any reader uses headings to get a rough idea of what the text content is about and will make the decision to read it further or not. We usually spend a very short time on a webpage and checking headings is part of our decision making process about staying or leaving. For that reason, working on a clear heading structure increases the chances of a reader staying and reading all of your content.

    Still not convinced? Okay, do you know that some people don't read webpages but listen to them? This is done with software call screen readers. This software provides ways to get fast access to given text content. Among the various technics they're using, providing an outline of the document by telling headings is the most common way to do this. If not, will they really listen to the whole document read out loud?

    Last but not least, using the outline of a document made of its headings is often used by search engines to guess the relevance of an article related to a given topic. Each heading level (from 1 to 6) as a relative weight in such guessing, means that a single level 1 heading is a bit more relevant than a few level 2 headings and so on.

    HTML Headings

    HTML provides a very convenient way to mark headings in HTML documents, those are the <h1>, <h2>, up to <h6> elements.

    Adding the top headline

    At the top of your document, write a short, gripping phrase inside <h1> tags:

    <h1>Empower yourself with these core Web skills</h1>

    If you think about your document as a tree, then the trunk is the <h1> heading. This implies that your document should only have one level 1 heading. Let's dig a bit into this as it's a common topic for (hot) discussion among web developers. Technically speaking, nothing prevents you from using several level 1 headings in your documents. However in a semantic way it is weird, as a level 1 heading is the foundation of a whole topic. If you feel the need to put several level 1 headings in your document, maybe that document is too long and should be split into several documents, each with its own level 1 heading.

    Another common argument for having a single level 1 heading in a document is its SEO impact. This is more debatable as the way search engine algorithms really work is a very well kept secret, however some observation tends to indicate that heading levels as an impact on SEO ranking and the more you use a given level heading the more it dilutes its impact on the ranking.

    Pro tip: A common mistake with the level 1 heading is to use it to highlight a web site logo or trademark. As it's meaningless information for your users nor neither the top most important information about your page content that you should take care in avoiding such practice.

    Adding sub-headings

    Unless your document is quite short, break it up into sections. Start each section with an <h2> heading:

    ...
<h2>Set up your document structure with HTML</h2>
...
<h2>Make your document legible and appealing with CSS</h2>
...

    If necessary, there are headings for subsections, from <h3> all the way to <h6>. That's to say, if you're starting a subsubsection with <h4> or lower, though, there is a high chance your post is too long and you should consider spliting up your document to simplify your reader's life.

    When you are creating sections and sub sections there are a few points that worth considering to improve your readers' experience.

    Keep you headings hierarchy consistent
    In plain terms that means that you should avoid skipping heading levels (e.g. putting a <h4> after a <h1> or <h2>). People, especially those using screen readers, could think something's missing.
    Use headings for their semantic, not their default style.
    HTML fundamentally provides text semantic. The default styling of a given HTML element is irrelevant as you can use CSS to change it. When it comes to headings, feel free to use the CSS font-size property to change the font size.
    Don't use headings elements for subtitles and taglines.
    One of the missing points of HTML is that it doesn't provide any specific element for subtitles and taglines. Some web developers tend to use headings elements to do this but it induces broken title hierarchy and bad accessibility. Even if there are some attempts to fix this with the proposed <hgroup> element, it is recommended to use regular <p> or <span> elements with a class.

    Special case: "dynamic" heading levels

    HTML 5 had introduced the <section> element to explicitly mark the various sections of an HTML document and each of these sections is expected to be followed by a heading of the proper level.

    Now, Let's suppose you're constantly moving whole sections around in your outline (it can happen often if your HTML is generated dynamically by your web server). Maybe you frequently decide that an <h3> section logically should form its own <h2> section. Or maybe you compile several webpages into one webpage to make an ebook. It can be pretty annoying to have to change all those heading levels.

    Provided your <section>s are nested logically, HTML 5 states that you can just begin each section with an <h1> and the <h1>s will still produce a valid document outline. If that's a very good idea, it suffers a few drawbacks: it makes your code less readable, and you can no longer style all <h1>s the same way if they aren't equally important.

    Please refer to Sections and Outlines of an HTML5 Document for more details.
